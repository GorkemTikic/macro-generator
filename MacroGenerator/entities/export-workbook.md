# Export Workbook Helper

Browser-side Excel generator for the FD Macro Generator admin dashboard. Produces a single styled `.xlsx` workbook with five sheets covering the analytics period selected in the UI. Replaces the earlier per-view CSV download buttons.

## Trigger

The dashboard header has a single "⬇ Export Workbook (.xlsx)" button. Clicking it:

1. Computes `since = Date.now() - sinceMs(range)` (range from the selector: 1d / 7d / 30d / 90d).
2. Calls `GET /admin/export?since=…` with the bearer token.
3. Passes the JSON response and a human range label (e.g. `"7 days (2026-04-19 → 2026-04-26)"`) to `downloadAnalyticsWorkbook()`.
4. ExcelJS builds the workbook in-browser and triggers a download named `fd-macro-analytics-YYYY-MM-DD.xlsx`.

No server-side file generation. No third-party upload. No external service sees the data.

## Backend endpoint — `GET /admin/export`

Auth: `Authorization: Bearer <ADMIN_TOKEN>` (same as other admin routes).

Query: `?since=<ms>&until=<ms>&limit=<n>` (defaults: last 30 days, 50 000 events).

Returns one JSON object:

```ts
{
  range: { since, until },
  overview:        { total_events, total_sessions, unique_devices, unique_ips, unique_countries },
  today_sessions:  number,
  tabUsage:        Array<{ tab: string;       count: number }>,
  topMacros:       Array<{ macro_type;        count: number }>,
  topSymbols:      Array<{ symbol;            count: number }>,
  lookupModes:     Array<{ mode;              count: number }>,
  fundingSymbols:  Array<{ symbol;            count: number }>,
  countries:       Array<{ country: string;   sessions: number }>,
  daily:           Array<{ day; sessions; events }>,
  errors:          Array<{ event_type; tab; props; ts }>,
  devices:         Array<{ device_id; country; total_events; sessions; first_seen; last_seen }>,  // up to 5 000
  events:          Array<{ id; event_type; session_id; device_id; country; tab; props; ts }>      // up to 50 000, ORDER BY device_id ASC, ts ASC
}
```

Events are pre-sorted by `device_id, ts` so the Device Journey sheet doesn't need to re-sort the entire stream.

## Public API


## Sheet builders

### `buildSummary`
- Frozen header row.
- Title + subtitle (period + generation timestamp).
- "Headline Metrics" block: total events, sessions, **estimated users (unique devices)**, hashed IPs, countries, today's sessions, plus derived ratios — events/session, sessions/device, total errors, error rate.
- Six side-by-side breakdowns laid out in two columns (A/B and C/D): top tabs, top macro types, top symbols, lookup modes, funding symbols, top countries.
- "In Plain English" block — narrative sentences generated by `buildPlainEnglish()`.

### `buildDeviceJourney`
- Frozen header + autofilter A1:H1.
- Events grouped by device. Each group is preceded by a dark separator row showing device id, country, event count, and time span.
- Within a group: zebra striping, all rows time-ordered ascending.
- Columns: Device, Session, Time (UTC), Country, Tab, Event, Action (human label), Key Details.
- "Action" column comes from `EVENT_LABEL` map; "Key Details" from `keyDetails()` which extracts `symbol`, `macro_type`, `mode`, `from→to`, `lang`, `message`/`error` from the props.

### `buildRawEvents`
- Frozen header + autofilter A1:I1.
- Columns: ID, Time (UTC), Event Type, Tab, Country, Session ID, Device ID, Props (parsed), Props (raw JSON).
- Both parsed and raw JSON kept side-by-side for audit value.

### `buildDevices`
- Frozen header + autofilter A1:I1.
- One row per device with derived columns: **Avg Events/Session**, **Most Used Tab**, **Last Activity** (human label of the most recent event).
- Derivations done in JS by indexing events by device, not in SQL.

### `buildReadyToShare`
- Two-column layout (narrow gutter + wide content column).
- Six sections: What this tool is used for · Adoption snapshot · Most used workflows · Example user behavior patterns · Business / operational value · Risks and current limitations.
- Sentences are populated from real metrics. Trend detection: compare sum of sessions in first half of `daily` vs. second half — `growing` if `secondHalf > 1.15 * firstHalf`, `declining` if `< 0.85 *`, otherwise `stable`.
- The Risks section explicitly admits the device-vs-user limitation.

## Event label map

Used by Device Journey ("Action" column) and Devices ("Last Activity" column).

| Event type | Human label |
|---|---|
| `page_view` | Opened the app |
| `tab_switch` | Switched tab |
| `lang_switch` | Changed language |
| `macro_generated` | Generated macro |
| `macro_error` | Macro generation failed |
| `grid_paste_used` | Pasted grid data |
| `lookup_query` | Ran price lookup |
| `lookup_error` | Lookup failed |
| `trailing_stop_checked` | Checked trailing stop |
| `funding_query` | Checked funding |
| `funding_error` | Funding lookup failed |
| `average_calc_run` | Used average calculator |
| `margin_view` | Viewed margin restrictions |
| `margin_refresh` | Refreshed margin data |
| `margin_error` | Margin lookup failed |
| `error` | Reported an error |

Unknown event types fall back to title-cased identifier.

## Style constants

- Header fill: `#1F2937` slate, white bold text, height 20.
- Section fill: `#F3F4F6` light grey.
- KPI cells: `#EFF6FF` light blue background, value in `#1D4ED8` bold.
- Zebra rows: `#FAFAFA`.
- Borders: thin `#E5E7EB` on all data cells.
- All data sheets: `views: [{ state: 'frozen', ySplit: 1 }]` and `autoFilter` on the header range.

## Bundle impact

`exceljs ^4.4.0` adds ~1 MB to the bundled JS (gzip ~370 KB). Vite warns about chunk size; not split because the workbook export is only used by admins and code-splitting was not deemed worth the complexity.
